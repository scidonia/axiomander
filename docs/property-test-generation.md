# Executable Test Generation from Contracts

Axiomander can generate runnable property-based tests directly from the
assert-based contracts already present in your Python source.  This turns
contracts from static proof obligations into executable CI gates.

## Motivation

Dafny autogenerates unit tests from specifications.  Axiomander currently
proves contracts statically -- every contract IR node has `to_coq()` /
`to_smt()` but no executable path.  However, contracts are already valid
Python expression ASTs (parsed in `contract_linter.py`), and the
verifier-only builtins (`implies`, `is_valid`, `is_shape`) are trivially
executable.  This makes an executable-test path cheap to build and adds
real value: contracts become runnable, not just provable.

---

## Three Levels

### Level A -- Hypothesis Generator (`py/oracle/test_gen.py`)

Reads a Python source file, extracts assert-based contracts via
`ContractLinter`, derives Hypothesis strategies from the IR, and emits a
runnable `pytest + hypothesis` test module.

**Key features:**

- **Strategy narrowing from preconditions.**  `assert x >= 0` becomes
  `st.integers(min_value=0)`.  `assert 0 <= x <= 100` becomes
  `st.integers(min_value=0, max_value=100)`.  Preconditions that cannot be
  absorbed into a strategy bound (e.g. `assert len(name) > 0`) become
  `assume(len(name) > 0)` inside the test body.

- **Type-driven strategies.**  Parameter type annotations drive the base
  strategy: `int` -> `st.integers()`, `float` -> `st.floats()`,
  `str` -> `st.text()`, `list` -> `st.lists()`, `bool` -> `st.booleans()`.

- **Derived parameters.**  `assert n == len(xs)` marks `n` as derived from
  `xs`; the generated test computes `n = len(xs)` in the body rather than
  generating it independently.

- **`old()` snapshot.**  Postconditions that reference `old(x)` produce an
  `_OldSnapshot(old_x=x)` binding before the function call, so the
  postcondition can be checked after the call.

- **Postconditions as asserts.**  Each postcondition IR node is rendered via
  `to_python()` and emitted as `assert <expr>  # <source text>`.

- **Exception postconditions.**  `RaisesExpr` nodes become
  `with pytest.raises(ExcType): ...` blocks.

**CLI usage:**

```bash
# Generate tests for all functions in a file
axiomander gen-tests py/examples/pydantic_contracts.py

# Generate tests for a single function
axiomander gen-tests py/examples/pydantic_contracts.py --function deposit

# Write output to a file
axiomander gen-tests py/examples/pydantic_contracts.py --output tests/test_contracts.py
```

**MCP tool usage:**

```json
{
  "tool": "gen_tests",
  "arguments": {
    "source": "<python source code>",
    "func_name": "deposit"
  }
}
```

**Example input:**

```python
def add(x: int, y: int) -> int:
    assert x >= 0
    assert y >= 0
    result = x + y
    assert result >= x
    return result
```

**Example output:**

```python
"""
Auto-generated property tests from assert contracts.
Source: <unknown>

Generated by: axiomander gen-tests
Do not edit -- regenerate with: axiomander gen-tests <file>
"""

import pytest
from hypothesis import given, assume, settings
from hypothesis import strategies as st

from oracle.contract_runtime import implies, is_shape, is_valid, re_match_pred, _OldSnapshot
# from <module> import add


@given(x=st.integers(min_value=0), y=st.integers(min_value=0))
def test_add_contracts(x, y):
    result = add(x, y)
    assert (result >= x)  # result >= x
```

### Level B -- pytest Plugin (`py/oracle/pytest_axiomander.py`)

Calls `mcp_server._verify_function` at pytest collection time.  The result
determines the test outcome:

| Verification result | pytest outcome |
|---------------------|----------------|
| `PROVED`            | PASS           |
| `COUNTEREXAMPLE`    | FAIL (prints SMT model) |
| `UNPROVED`          | xfail (expected failure) |
| `LEVEL3`            | xfail (LLM oracle needed) |

This makes formal verification a CI gate: a function that was previously
proved but whose contracts have been weakened will now fail CI.

**Usage:**

Register the plugin in `conftest.py`:

```python
# conftest.py
pytest_plugins = ["oracle.pytest_axiomander"]
```

Then mark functions for verification:

```python
import pytest

@pytest.mark.axiomander_verify(source_file="py/examples/demo.py", func_name="add")
def test_add_verified():
    pass  # body is replaced by the plugin
```

Or use the auto-collection mode, which discovers all `.py` files in the
project and generates a verification test for each function that has
contracts:

```bash
pytest --axiomander-verify py/examples/
```

### Level C -- Counterexample to Regression Test

When the SMT solver returns a counterexample (a concrete assignment of
variable values that violates a postcondition), Axiomander can emit a
concrete failing pytest case with the exact inputs.  This turns a failed
proof into a permanent executable regression test.

**API:**

```python
from oracle.test_gen import counterexample_to_test

rendered = counterexample_to_test(
    func_name="add",
    params=["x", "y"],
    counterexample={"x": 0, "y": -1},
    postcond_src="result >= x",
)
```

**Output:**

```python
def test_add_regression_counterexample():
    """Regression test: SMT counterexample found by axiomander.
    Violated postcondition: result >= x
    """
    x = 0
    y = -1
    # This call should expose the contract violation:
    result = add(x, y)
    # Postcondition that was violated: result >= x
    # TODO: assert the correct behaviour here
```

The generated test is appended to the test file and committed.  It serves
as a permanent record of the violation and a regression guard once the bug
is fixed.

## Public API Reference

### `generate_tests(source, func_name=None, module_path="") -> str`

Generate a complete Hypothesis test module from source contracts.

| Parameter | Type | Description |
|-----------|------|-------------|
| `source` | `str` | Python source code with assert contracts |
| `func_name` | `str \| None` | If given, generate tests only for this function |
| `module_path` | `str` | Path to source file (used for import line) |

Returns a complete, importable `pytest + hypothesis` test module as a string.

### `extract_function_contracts(source, func_name) -> FunctionContracts`

Parse a Python source string and extract contracts for a single function.

Returns a `FunctionContracts` dataclass with:
- `func_name`: function name
- `params`: ordered list of parameter names
- `param_types`: dict mapping param name to type string
- `preconditions`: list of `Expr` IR nodes (from precondition asserts)
- `postconditions`: list of `Expr` IR nodes (from postcondition asserts)
- `exception_postconditions`: list of `RaisesExpr` nodes
- `old_bindings`: dict mapping `old_x` -> `x` for `old()` references
- `precond_sources` / `postcond_sources`: raw source text for each contract

### `counterexample_to_test(func_name, params, counterexample, postcond_src="") -> str`

Emit a concrete regression test from an SMT counterexample dict.

| Parameter | Type | Description |
|-----------|------|-------------|
| `func_name` | `str` | Name of the function that failed verification |
| `params` | `list[str]` | Ordered list of parameter names |
| `counterexample` | `dict[str, int]` | Variable -> value mapping from SMT model |
| `postcond_src` | `str` | Optional: postcondition source text that was violated |

### `ParamStrategy`

Dataclass representing a Hypothesis strategy for a single parameter.

| Field | Type | Description |
|-------|------|-------------|
| `name` | `str` | Parameter name |
| `py_type` | `str` | `"int"`, `"float"`, `"str"`, `"list"`, `"bool"` |
| `min_value` | `int \| None` | Lower bound (for integers/floats) |
| `max_value` | `int \| None` | Upper bound (for integers/floats) |
| `min_size` | `int \| None` | Minimum length (for strings/lists) |
| `max_size` | `int \| None` | Maximum length (for strings/lists) |
| `derived_from` | `str \| None` | If set, this param is computed from another |
| `derived_expr` | `str \| None` | Python expression to compute derived value |

Call `.to_hypothesis()` to get the Hypothesis strategy string.

## Contract Runtime (`py/oracle/contract_runtime.py`)

The generated tests import `oracle.contract_runtime` for the executable
counterparts of the Coq/SMT-only predicates:

| Function | Signature | Description |
|----------|-----------|-------------|
| `implies(a, b)` | `(bool, bool) -> bool` | Logical implication: `(not a) or b` |
| `is_shape(obj, model_type)` | `(Any, str) -> bool` | Structural shape check via shape registry |
| `is_valid(obj, model_type)` | `(Any, str) -> bool` | Shape + field constraint check |
| `re_match_pred(subject, pattern)` | `(str, str) -> bool` | Full-string regex match |
| `_OldSnapshot(**kwargs)` | `(**Any) -> _OldSnapshot` | Immutable pre-call value snapshot |

All functions are pure and side-effect-free.  `is_shape` and `is_valid` are
conservative: if the model type is not registered in the shape registry, they
return `True` rather than failing.

## Running the Generated Tests

```bash
# Install hypothesis
uv pip install hypothesis

# Generate tests
axiomander gen-tests py/examples/pydantic_contracts.py \
    --output py/tests/test_pydantic_contracts_generated.py

# Run them
uv run pytest py/tests/test_pydantic_contracts_generated.py -v
```

## Design Notes

### Why assert-based contracts?

Axiomander's contracts are plain Python `assert` statements -- no decorators,
no imports.  The `ContractLinter` classifies them by position:

- **Precondition**: assert at function start, before any non-assert code
- **Postcondition**: assert immediately before a `return` statement
- **General**: everything else (loop invariants, mid-function checks)

Only preconditions and postconditions are used for test generation.

### Why Hypothesis?

Hypothesis is the de-facto standard for property-based testing in Python.
Its `@given` decorator + strategy combinators map directly onto the
precondition/postcondition structure of Hoare-logic contracts:

- Preconditions -> strategy bounds + `assume()` filters
- Postconditions -> `assert` checks after the call

### Strategy narrowing vs assume()

Strategy narrowing (absorbing bounds into `min_value`/`max_value`) is
preferred over `assume()` because Hypothesis can generate values more
efficiently when the strategy is already bounded.  `assume()` causes
Hypothesis to discard generated values that don't satisfy the filter,
which can be slow for tight constraints.

The generator absorbs simple `param op literal` constraints into the
strategy and falls back to `assume()` for complex expressions (e.g.
`len(name) > 0`, `n == len(xs)`).

### Relationship to formal verification

Test generation is complementary to, not a replacement for, formal
verification:

- **Formal verification** (Levels 1-3 of the proof pipeline) proves the
  contract holds for *all* inputs.
- **Property-based testing** (this feature) checks the contract holds for
  *many* randomly generated inputs.

A function that passes Hypothesis tests but fails formal verification has a
contract that is too weak to prove -- the tests are passing by luck.  A
function that fails Hypothesis tests definitely has a bug.

The Level B pytest plugin bridges the two: it runs formal verification at
collection time and reports the result as a test outcome, making the proof
pipeline a first-class CI gate.
